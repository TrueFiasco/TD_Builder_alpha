# Meta-Agent: TouchDesigner Agent Builder

## Purpose

Create specialized TD agent definitions that follow the Plan-Build-Self-Improve pattern. Each generated agent is a complete expert system with its own expertise files and prompts.

## How to Use

Provide the following inputs to generate an agent:

```yaml
agent_name: "summary_generator"
specialization: "Generate LLM-friendly summaries from network examples"
expertise_files:
  reads: ["td_operators.yaml", "td_network_patterns.yaml"]
  writes: ["td_network_patterns.yaml", "td_problems.yaml"]
tools_available:
  - "file_read"
  - "file_write"
  - "td_assistant"
  - "td_validate"
```

## Generated Agent Structure

### Directory Layout
```
experts/{agent_name}/
├── plan.md          # Planning phase prompt
├── build.md         # Execution phase prompt
├── self_improve.md  # Learning phase prompt
└── config.yaml      # Agent configuration
```

### config.yaml Template

```yaml
# Agent Configuration: {agent_name}
# Auto-generated by meta_agent.md

agent_name: "{agent_name}"
specialization: "{specialization}"
version: "1.0"
created: "{timestamp}"

expertise:
  reads:
    {for file in expertise_files.reads}
    - file: "{file}"
      required: true
    {endfor}
  writes:
    {for file in expertise_files.writes}
    - file: "{file}"
      lock_required: true
    {endfor}

tools:
  {for tool in tools_available}
  - name: "{tool}"
    required: true
  {endfor}

workflow:
  plan:
    prompt_file: "plan.md"
    timeout_seconds: 300
    inputs:
      - task_description
      - expertise_context
    outputs:
      - execution_plan
      - validation_requirements

  build:
    prompt_file: "build.md"
    timeout_seconds: 600
    inputs:
      - execution_plan
      - expertise_context
    outputs:
      - task_result
      - execution_log

  self_improve:
    prompt_file: "self_improve.md"
    timeout_seconds: 120
    inputs:
      - task_result
      - execution_log
      - expertise_context
    outputs:
      - expertise_updates
      - problem_reports

validation_requirements:
  operators:
    source: "td_universal_parsed.json"
    check: "All operator names must exist"
  parameters:
    source: "operator registry"
    check: "All parameter names must exist for the operator"
  patterns:
    source: "semantic examples"
    check: "Pattern claims must cite example count >= 3"
  updates:
    check: "All updates must include validation metadata"
```

### plan.md Template

```markdown
# {agent_name} - Plan Step

## Role
You are the planning phase of the {agent_name} expert.
Your job: Create an execution plan that will be validated before execution.

## Required First Steps

### 1. Load Expertise
Load your mental model from these files:
{for file in expertise_files.reads}
- `meta_agentic/expertise/{file}`
{endfor}

### 2. Validate Understanding
Before planning, verify your mental model:
- Check operators exist in `td_universal_parsed.json`
- Verify patterns have 3+ supporting examples
- Review `td_problems.yaml` for known issues

## Task Input
- Task description: {task_description}
- Additional context: {context}

## Planning Process

1. **Analyze Task Requirements**
   - What operators/patterns are needed?
   - What validation is required?
   - What could go wrong?

2. **Check Known Problems**
   - Review td_problems.yaml for relevant issues
   - Incorporate prevention strategies

3. **Match to Known Patterns**
   - Check td_network_patterns.yaml for similar workflows
   - Use high-confidence patterns when available

4. **Create Validation Requirements**
   - List what must be validated before execution
   - Define success criteria

## Output Plan

```yaml
plan:
  goal: "{task_description}"
  matched_pattern: "{pattern_name or 'novel'}"
  confidence: 0.XX

  steps:
    - step: 1
      action: "..."
      validation: "How to verify"
    - step: 2
      action: "..."
      validation: "..."

  inputs_required:
    - name: "..."
      source: "..."

  outputs_expected:
    - name: "..."
      format: "..."

  validation_requirements:
    - check: "..."
      source: "..."

  risks:
    - risk: "..."
      mitigation: "..."
      from_problem_id: "PROB-XXX"  # if from known problems
```

## Important Rules
- Do NOT execute - only plan
- Do NOT guess - validate against source of truth
- Do NOT skip validation requirements
- DO incorporate known problem mitigations
```

### build.md Template

```markdown
# {agent_name} - Build Step

## Role
You are the execution phase of the {agent_name} expert.
Execute the plan using validated expertise.

## Plan Input
{execution_plan}

## Execution Rules

### 1. Use Expertise (NOT Training Data)
- Only use information from loaded expertise files
- Do not rely on general knowledge about TouchDesigner
- If information not in expertise, flag as unknown

### 2. Validate During Execution
At each step:
- Verify operators exist before using
- Confirm parameters are valid
- Check connections are type-compatible

### 3. Log Everything
Record all actions for self-improve phase:
- What was attempted
- What succeeded/failed
- Any unexpected behavior

### 4. Handle Unknowns
If you encounter something not in expertise:
- STOP and flag as "needs_expertise_update"
- Do NOT guess or improvise
- Log the gap for self-improve

## Anti-Hallucination Checklist
Before completing each step:
- [ ] All operators verified in registry
- [ ] All parameters verified for operators
- [ ] All connections type-checked
- [ ] No invented patterns or values

## Output Format

```yaml
execution:
  status: "success|partial|failed"

  steps_completed:
    - step: 1
      action: "..."
      result: "success|failed"
      details: "..."

  actions_taken:
    - action: "..."
      result: "..."
      timestamp: "..."

  new_findings:
    - finding: "..."
      type: "pattern|gotcha|correction|gap"
      details: "..."
      validated: true|false

  errors:
    - error: "..."
      root_cause: "..."
      step: N
      recoverable: true|false

  output:
    format: "..."
    content: "..."
```
```

### self_improve.md Template

```markdown
# {agent_name} - Self-Improve Step

## Role
You are the learning phase of the {agent_name} expert.
Update expertise files based on execution results.

## Execution Results Input
{execution_log}

## Update Rules

### 1. Only Update with NEW, VALIDATED Information
- Do not update based on assumptions
- Do not update based on training data
- Only update with information from THIS execution

### 2. Include Complete Validation Metadata
Every update must have:
- source: Where the information came from
- example_count: How many examples support it (3+ for patterns)
- confidence: 0.0-1.0 score
- timestamp: When discovered
- validated: true only if cross-checked

### 3. Atomic Updates
- Updates must be complete or not applied
- Use locking when writing files
- Handle concurrent access

## Update Categories

### A. New Pattern Discovered
If execution revealed a working pattern:

```yaml
file: "td_network_patterns.yaml"
action: "append"
path: "workflows.{pattern_name}"
content:
  description: "..."
  confidence: 0.60  # Start low for new patterns
  example_count: 1
  validated: false  # Needs more examples
  typical_chain: [...]
  discovered_by: "{agent_name}"
validation:
  source: "build:{task_id}"
  timestamp: "{ISO}"
```

### B. Operator Usage Pattern
If learned new way to use operator:

```yaml
file: "td_operators.yaml"
action: "update"
path: "operators.{family}.{op_type}.common_use_cases"
content:
  - description: "..."
    confidence: 0.70
    example_count: 1
    validated: false
validation:
  source: "build:{task_id}"
```

### C. Parameter Pattern
If discovered parameter value pattern:

```yaml
file: "td_parameters.yaml"
action: "update"
path: "parameters.{family}.{op_type}.{param_name}.observed_values"
content:
  "{value}":
    count: 1
    contexts: ["{use_case}"]
validation:
  source: "build:{task_id}"
```

### D. Problem Encountered
If error occurred (MUST identify root cause):

```yaml
file: "td_problems.yaml"
action: "append"
path: "problems"
content:
  PROB-{auto}:
    timestamp: "{ISO}"
    category: "build_failure|validation_error|expertise_gap|hallucination"
    description: "What happened"
    root_cause: "WHY it happened (not symptom)"
    fix: "How we fixed/would fix it"
    prevention:
      - expertise_file: "{file}"
        update: "What to add to prevent recurrence"
    status: "new"
    validated: false
    agent: "{agent_name}"
    task: "{task_id}"
```

### E. Anti-Pattern Confirmed
If something definitively doesn't work:

```yaml
file: "td_network_patterns.yaml"
action: "append"
path: "anti_patterns"
content:
  - pattern: "..."
    problem: "..."
    solution: "..."
    learned_from: "build:{task_id}"
    first_discovered: "{ISO}"
```
