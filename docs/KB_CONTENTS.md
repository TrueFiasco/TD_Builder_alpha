# KB — knowledge base contents

The `KB/` folder is TD Builder's knowledge base. The large/generated artifacts are distributed via a
GitHub Release (fetched by `scripts/fetch_vector_db.py`) rather than committed, to keep the repo
lean. `KB/README.md` is the short, in-folder version of this document.

| File / dir | What it is | Ships how |
|---|---|---|
| `operators.json` (~34 MB) | Enriched operator specs (the source of truth for validation + lookups) | in the bundle |
| `knowledge_graph_enhanced.gpickle` (~9 MB) | The knowledge graph (operators, params, examples, patterns) | in the bundle |
| `vector_db/` (~41 MB) | ChromaDB vector store — 6,447 condensed pointer chunks, embedded with **all-MiniLM-L6-v2** (local, key-free) | **fetched** via `scripts/fetch_vector_db.py` (not in git) |
| `lexical_index/bm25.pkl` (~8 MB) | BM25 lexical index backing hybrid (dense + sparse) retrieval | in the bundle |
| `models/ms-marco-MiniLM-L-6-v2/` (~87 MB) | Bundled cross-encoder reranker (offline, key-free) | in the bundle |
| `sources.lock.json` | Build provenance (pinned source revisions) | in the bundle |
| `wiki_supplemental/` | Hand-authored how-to guides (`Write_a_GLSL_TOP.md`, `Write_GLSL_POPs.md`, `Write_a_GLSL_Material.md`) | **in git** (committed — small markdown) |
| `docked_dats.json` | Per-op docked-DAT specs used by the builder | **in git** (committed) |
| `sources/` | Snippet/palette source captures (KB-rebuild inputs only) | **NOT shipped** — dev-only; rebuild from the dev repo if needed |
| `manifest.json` | KB bundle metadata (sections, chunk histogram, retrieval stack) | **in git** (committed) |
| `kb_receipt.json` | Load-time trust receipt (sha256 of the pickled artifacts; see below) | written locally by fetch/build — never committed |

## Pickle integrity (load-time trust boundary)
`pickle.load` of a tampered file is arbitrary code execution, so the runtime verifies the two
pickled artifacts (`lexical_index/bm25.pkl`, `knowledge_graph_enhanced.gpickle`) **before**
unpickling: the bytes must sha256-match `KB/kb_receipt.json` (written by `fetch_vector_db.py` at
verified-extract time and by the `kb_build` writers) or the committed `artifact_sha256` pins in
`scripts/vector_db_release.json` (`MCP/server_core/kb_integrity.py`). Refused artifacts degrade
loudly — dense-only retrieval / graph features off — instead of loading. Bless a self-built KB with
`python scripts/receipt_kb.py`; `TD_BUILDER_TRUST_KB=1` bypasses with a loud warning (dev only).

## Retrieval stack (v0.2)
Search runs a hybrid pipeline: dense (`all-MiniLM-L6-v2`) + BM25 → RRF(k=60) → operator-aware router →
cross-encoder rerank → calibrated score-floor + dedup (`MCP/server_core/search/retrieval_stack.py`).
The v0.1.x `graphrag.json` is gone — those chunks now live as condensed pointer chunks in `vector_db/`
and hydrate from `operators.json` at query time, which is what shrank the store from ~34k docs to 6,447.

## Embedding model — do not change
The vector store was built with `all-MiniLM-L6-v2`. Query-time embedding **must** use the same
model (it does, by default). Switching to a cloud embedding provider would mismatch the store and
break search. This is what keeps the system key-free. The bundled `models/` reranker is a
*cross-encoder*, not the embedder.

## Rebuilding the KB (developers)
The vector store + graph + lexical index + reranker are regenerated by the `kb_build/` pipeline (kept in
the dev repo, not shipped). End users never need to rebuild — they fetch the pre-built
`td_builder_kb_v0.2.0.zip` bundle.
