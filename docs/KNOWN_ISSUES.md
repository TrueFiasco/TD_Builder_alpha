# Known Issues ŌĆö V0.2.0

Documented limitations, not blockers. The 22-check acceptance + smoke gate passes; KB search is
local / key-free.

| Area | Issue | Status / workaround |
|---|---|---|
| **Palette components** | Per-operator `palette` fields build an **external-tox placeholder** that loads the component from the **user's own TD install** at open time (`app.samplesFolder` expression + `subcompname`) ŌĆö no Derivative content is bundled. 277 items registered in `KB/palette_components.json`; wires to/from the placeholder survive loading (TD-native `compinputs` + inner-out-op serialization). | **Works.** Caveats: components mirror the palette of the TD version they load on (registry harvested on 2025.32820); an item missing from the user's install loads empty (no crash). Design-level `{"palette": ŌĆ”}` keys are rejected with a pointer to the per-operator form. For unregistered `.tox` files use `external_tox`. |
| **Component as a data source** | A component is never itself a data source ŌĆö what it *outputs* lives on its inner `out` op, separate from what it *displays*. The inner out op's **actual name** matters (comps use custom names like `valueOut`/`out_bins`; it is not always `out1`) ŌĆö get it from `expand_toe_file(mode='summary')`'s `manifest`. Residual **comp-idiom (BUG-4)** caveat, guidance-only: an expression like `op('comp')['sin']` errors at runtime (`baseCOMP not subscriptable`) ŌĆö write `op('comp/<outOp>')['sin']` instead. | **`external_tox`/palette wire-drop: FIXED** on the v0.2.1 line (`7bac681` bridge pre-pass + provenance-aware resolution; merged `531a997`). The builder manifest-parses the referenced `.tox` at build time; bare `{"from"/"to": "comp"}` wires auto-resolve when the component has exactly one inner out/in op (or the registry entry is live-harvested index-authority), otherwise the build **fails loudly naming the candidate inner ops** (never a silent drop). **W3a adds the static half:** `td_validate` now carries a `component_wiring` stage that flags a bare-component *source* before a build runs, with the same vocabulary + candidate list (`external_tox` sources defer to the builder's build-time manifest check). Explicit inner names (`comp/<outOp>`, `comp/<inOp>`) always work; wrapper-style `.tox`es need `parameters.subcompname`. The comp-idiom (BUG-4) caveat above is unaffected and **remains disclosed**. |
| **BASIC-mode build** | Building a network from scratch (no parsed source) emits parameter-format warnings; some expression modes are approximate. | The **LOSSLESS** round-trip (parse an existing `.toe.dir` ŌåÆ rebuild) is byte-accurate ŌĆö use it when you have a source. |

| Live tools | Require TouchDesigner running + `MCP/td-webserver/mcp_webserver_base.tox` imported (port 9981). | Without TD, the 21 `td-builder-live` tools return a clear "not running" message. |
| **Live-server auth** | The WebServer DAT binds all interfaces (cannot be loopback-only). Requests need `Authorization: Bearer <token>`; missing/invalid ŌåÆ **401** with a self-describing hint. | TD generates the token on first boot at `~/.td_builder/api_token` and prints only the path. Same machine = zero config (the client reads it). Remote client = copy the value, set `TD_API_TOKEN` in the client env. Rotate = delete the file + **restart TD**. `GET /api/td/server/td` stays open (health). See `MCP/COMM_LAYER.md`. |
| **Live-server transport** | Token + payloads travel **plaintext HTTP** (no TLS this release). | Token auth is the LAN control; for untrusted networks tunnel via SSH/VPN or stay on `127.0.0.1`. Debug logging is OFF by default (`TD_API_DEBUG=1`); capture payloads are never logged. |
| **`execute_python_script` can hang TD (via user code)** | A script YOU pass can call `ui.*` (messageBox/chooseFile/ŌĆ”) or `project.save()`, which raise a modal on TD's single main thread and freeze the live connection (~60 s, no timeout rescue). The committed server code is itself dialog-free (CI-locked by `tests/unit/test_no_save_dialog_guard.py`), and the KB/skill/tool-description warn against it ŌĆö but the exec namespace injects the live `project`/`ui` singletons (dropping them is bypassable via `import td` and would break legitimate `project.folder`/`ui.status` reads, so it is kept). | `execute_python_script` is `DESTRUCTIVE`-annotated (blocked under auto-mode). To checkpoint, call **`save_td_project`** (dialog-proof) ŌĆö never `project.save()` inside a script. |
| **Restore-point path leak (tokenless)** | The tokenless health route `GET /api/td/server/td` returns `restorePoint.path` ŌĆö an absolute path under `~/.td_builder/restore_points/`, which leaks the **local username** to any unauthenticated LAN caller. | **Accepted + documented** (marginal: the caller already reached a machine-identifying endpoint; consistent with the plaintext-transport threat model above). The full snapshot metadata is otherwise only on the authenticated `get_mutation_status`. |
| **Multi-expert orchestration** | The `get_expert_prompt` prompts describe a plan ŌåÆ build ŌåÆ critic pipeline, but this release has no automated orchestrator ŌĆö you fetch and run each expert prompt individually. | **Deferred to a later release.** Prompts now route all knowledge through the MCP tools; the cross-expert pipeline is manual for now. |
| **Self-improvement / curated expertise** | The experts' `self_improve` phase and the 18 curated `Agents/expertise/*.yaml` are not wired into the runtime (only `td_network_building.yaml` is read by code). | **Deferred to a later release.** Those YAMLs plus three currently-unused engine modules (`engine/writers/lossless_writer.py`, `server_core/search/__init__.py`, `Agents/experts/__init__.py`) are kept for a future "connect in or remove" decision. |

**No API key anywhere** ŌĆö KB semantic search uses a local embedding model. The first KB-dependent
call warms the knowledge base (~1ŌĆō2 min, one-time), then every call is fast.
