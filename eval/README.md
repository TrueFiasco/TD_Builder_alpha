# TD Builder v0.2 — Phase 0 evaluation harness

The gate every later phase of the KB/retrieval redesign must beat. It measures the
**current shipped search** offline and in-process, with a swappable backend so the
**same harness is reused unchanged through Phase 2/3**.

See the plan §3 (baseline), §5 (ground-truth + name-integrity), §9 Phase 0, §10
(verification): `plan-mode-task-redesign-serialized-kite.md`.

## Files

| File | What it is |
|---|---|
| `labeled_queries.jsonl` | 78 hand-labeled queries: 12 × 6 scored categories (operator_lookup, parameter, palette_discovery, howto, python, build_instruction) + 6 `negative` (abstention). Each row `{id, query, category, relevant_predicate}` (negative rows use `expect_no_match` instead). |
| `predicates.py` | Ground-truth loader, predicate matcher, name-integrity gate (reusable). |
| `run_eval.py` | Driver: builds the shipped adapter, runs queries, computes metrics, writes reports. |
| `baseline.json` | The captured current numbers (the gate). Regenerated by `run_eval.py`. |
| `tool_coverage.py` | Phase-0.5 driver for the OTHER KB tools (get_operator_info / get_parameter_detail / find_operator_examples / find_operator_combination / get_network_patterns). |
| `tool_baseline.json` | Captured tool-behaviour baseline. Regenerated by `tool_coverage.py`. |
| `gen_coverage.py` | Phase-0.6 **coverage-tier** generator: mints a stratified, ground-truth-derived breadth set (`coverage_queries.jsonl`), seedable + offline. See "Coverage tier" below. |
| `coverage_report.py` | Joins `run_eval` per-query scores with the coverage `gen` metadata → per-family / per-chunk-type / per-param-kind breakdowns (`run_eval` aggregates by category only). |
| `coverage_queries.jsonl` | A sample coverage set (seed 0); every item carries a `gen.ambiguity` tag. Regenerated/rotated by `gen_coverage.py --seed N`. |
| `paraphrase_cache.json` | Optional dev/build-time LLM paraphrases (naturalness only; gold never changes), keyed `'<seed>:<id>'`. Used only with `--paraphrase`; default OFF. |
| `minimal_cover.py` | Greedy partial set-cover → the FEWEST tests for a coverage FLOOR (≥80% ops + ≥75% per section). Writes `coverage_minimal.jsonl`. |
| `build_sweet16.py` | Curated GOLD queries for Derivative's official **Sweet 16** (CHOP/TOP/SOP — the only families with one). Writes `sweet16_queries.jsonl` (48). |
| `sweet16_queries.jsonl` | 48 `operator_lookup` gold queries (16 each for CHOP/TOP/SOP), `gen.tier="sweet16"`. Scorable by `run_eval.py`. |

Staged outputs (NOT committed; reviewer copies) land in
`New KB build/Output/eval/`: `baseline.json`, `BASELINE.md`, `details.json`.

## Run

```bash
py -3.11 eval/run_eval.py                  # both backends, median of 3 trials; writes baseline.json + staged reports
py -3.11 eval/run_eval.py --backend enhanced --repeats 5
py -3.11 eval/run_eval.py --repeats 1      # quick single trial
py -3.11 eval/run_eval.py --kb "C:/path/to/new/KB" --compare eval/baseline.json   # measure a rebuilt KB, diff vs baseline
```

Phase-1 flags: `--kb <rebuilt KB>` points the search at a new build; `--compare <prior baseline.json>`
prints per-category recall/MRR/nDCG deltas + name-integrity delta. **Ground truth is resolved
independently of `--kb`** (so the name-integrity gate stays authoritative when measuring a rebuilt
KB) — override with `--gt-operators` / `--gt-types`. `--score-floor` (default 0.2) sets the
negative-query abstention threshold.

Fully offline (local MiniLM + Chroma, no API, cache disabled; the harness forces
`HF_HUB_OFFLINE` + single-thread BLAS for reproducible embeddings). The worktree has
no KB (it's gitignored), so the harness auto-resolves the **main-tree** `KB/` by
stripping the `.claude/worktrees/<name>` tail; override with `--kb` or `EVAL_KB_ROOT`.
Code is imported from this repo's `MCP/server_core`; only the KB **data** comes from
the main tree.

### Reproducibility (`--repeats`)

Embeddings, DB content, and the on-disk HNSW index are all verified byte-stable, but
**ChromaDB's HNSW approximate search is not bit-deterministic across process loads** —
the in-memory graph traversal returns slightly different borderline neighbors, so ~1
rank-5/6 result can flip run-to-run. The jitter is fixed *per process* (in-process
repeats agree), so `--repeats N` runs **N independent subprocesses** (fresh index load
each) and reports the **median** per metric plus the observed band. Measured jitter is
confined to **operator_lookup** (recall@5 0.83–0.92); palette/howto/parameter/python and
`unresolved=0` are invariant. Default `--repeats 3`; use 5 for the canonical capture.

## Backend swap (A/B) — the whole point

`run_eval.py` imports the real `UnifiedSearchAdapter` and calls `.search()` — the exact
call the live `hybrid_search` MCP tool makes.

- `--backend enhanced` → `use_legacy=False`: the **shipped path** and the slot Phase 2
  plugs its new stack into (`_enhanced_search`). **This is THE baseline.**
- `--backend legacy` → `use_legacy=True`: the frozen `HybridGraphRAG` reference; it does
  **not** change as `_enhanced_search` is edited.

Today both produce identical `semantic_results` (same dense MiniLM `TDDocSearch` over
`td_unified`), so their ranking metrics match; they diverge once Phase 2 lands. Re-run
`--backend enhanced` after each Phase-1 section rebuild and after the Phase-2 stack
change to get the attributable delta.

## Metrics

- **recall@k** — hit-rate / success@k: 1 if ≥1 relevant chunk in top-k, else 0.
  (Predicates label an open set, not an enumerated gold set, so this is what the plan's
  "recall@5" table reports.)
- **MRR** — 1 / rank of the first relevant chunk.
- **nDCG@n** — binary-gain DCG over top-n ÷ ideal DCG normalized by `min(corpus gold
  count, n)`. The gold count comes from a one-off full chunk-store predicate scan per
  trial (`_gold_counts`). NOTE (2026-07-01): the formula previously normalized by the
  *retrieved*-relevant count, which scored 1.0 for any query whose few hits ranked
  first regardless of misses — absolute nDCG values in a `baseline.json` captured
  before this fix are inflated and should be recaptured (recall@k and MRR are
  unaffected; cross-arm deltas within one baseline remain internally consistent).

## Labels survive re-chunking

A `relevant_predicate` is a disjunction of conjunctive clauses over **STABLE** identity
fields only — never chunk ids:

```json
{"clauses": [{"python_class_any": ["noiseTOP_Class"]},
             {"op_name_any": ["Noise TOP"]}]}
```

A chunk is relevant iff it matches **any** clause; a clause matches iff **all** its keys
match. Supported keys (see `predicates.FIELD_MATCHERS`): `type_any`, `family_any`,
`store_any`, `python_class_any`, `class_any`, `method_any`, `param_code_any`,
`meta_name_any`, `term_any`, `op_name_any` (operator-name, normalized across
name/operator/operator_name), and `text_contains_any` (content substring). Most palette
and howto predicates also carry the **future** v0.2 chunk types (`block_overview`,
`recipe`, `guide`, …) so the same labels score the rebuilt KB without edits.

Labels were hand-verified against the live MCP `hybrid_search` and the ground-truth
files (`KB/operators.json`, `operator_ground_truth/operator_types.json`): operator
`python_class` tokens, parameter codes, palette block names, and `class_method` /
`python_pattern` names all exist as written.

## Tool-coverage harness (`tool_coverage.py`, Phase 0.5)

`run_eval.py` gates the *ranking* tool (`hybrid_search`). `tool_coverage.py` gates the
**other** KB-dependent MCP tools — the hydration/lookup + pattern tools the builder uses
once it knows an operator — so tool regressions/fixes are visible as the KB is rebuilt. It
drives the real engine the MCP handlers delegate to (`UnifiedGraphQuery`), offline +
in-process, reusing the same labels, and prints a compact scorecard (details → disk).

```bash
py -3.11 eval/tool_coverage.py            # writes tool_baseline.json + New KB build/Output/eval/TOOL_COVERAGE.md
```

Checks: `get_operator_info` (resolve / canonical name / has-params), `get_parameter_detail`
(param found, default present, **default matches the `operator_ground_truth` live-TD
capture**), `find_operator_examples` (operators with ≥1 example), `find_operator_combination`
(intent operator-sets, natural tokens), `get_network_patterns` (inventory + `DAT:text`
readMe-pollution count). Deterministic (graph/dict lookups, no HNSW jitter). The defaults
check is the headline: it tells you which param defaults the tool already gets right (so §6.2
re-grounding can target the long tail) vs wrong.

Captured baseline: get_operator_info 15/15 resolve correctly; get_parameter_detail returns the
correct live-TD default on **12/13 verifiable** params — the scalar defaults are right (the
structured `default` field is good; only the description prose has `⊞` artifacts), but **tuplet
params break**: Lag CHOP's `lag` is really `lag1`/`lag2`=0.2 and `get_parameter_detail('Lag CHOP',
'lag1')` returns *not found* (the harness probes the real tuplet code via `ParamDefaults.resolve`).
find_operator_examples 14/15; find_operator_combination 4/8 intents with natural tokens (misses are
TD short-OPType token brittleness — comp/geo/cam — not absent data); get_network_patterns 45/45
readMe-polluted. All cross-verified against the live MCP server. Phase-1 flags: `--gt-operators` /
`--gt-types` / `--params-dir` (stable, independent of `--kb`).

## Coverage tier (`gen_coverage.py`, Phase 0.6)

The frozen 78 (`labeled_queries.jsonl`) touches only **23 distinct operators (3.5% of 663)**;
POP (100 ops) and MAT (13) are never tested, and ~13 of ~16.7k params appear — so "aggregate
0.93" describes a CHOP/TOP-heavy slice and is silent about the other 96.5%. The coverage tier
fills that hole: a **stratified, ground-truth-derived BREADTH set** across all 7 families, the
main §6 chunk types, and the parameter kinds.

```bash
py -3.11 eval/gen_coverage.py --seed 0                     # -> eval/coverage_queries.jsonl (+ static KB check)
py -3.11 eval/run_eval.py --backend enhanced --repeats 1 \
    --kb "C:/.../New KB build/Output/KB" --queries eval/coverage_queries.jsonl \
    --gt-operators "C:/.../Output/KB/operators.json" \
    --gt-types "C:/.../Resources/operator_ground_truth/operator_types.json" \
    --stage-dir "C:/.../Output/eval/coverage" --out "C:/.../Output/eval/coverage/coverage_baseline.json"
py -3.11 eval/coverage_report.py --seed 0 \
    --queries eval/coverage_queries.jsonl \
    --details  "C:/.../Output/eval/coverage/details.json" \
    --baseline "C:/.../Output/eval/coverage/coverage_baseline.json" \
    --out      "C:/.../Output/eval/coverage/COVERAGE.md"
```

**Two hard guarantees per item (Phase-0.6b hardening — see `COVERAGE_AUDIT.md`).**
1. **Gold DERIVED FROM GROUND TRUTH**, never LLM-guessed, and it RESOLVES against ground truth (the
   run reports **0 unresolved**). No answer leak — the operator / comp / term / method name never
   appears in the query (single-word capability words like "noise"/"blur" are kept; FULL name /
   class / multi-word identifying core stripped at word boundaries).
2. **Gold is the UNIQUE / clearly-best answer.** A uniqueness gate classifies every candidate into
   three outcomes (the "enlarge a rectangle" problem) and tags each item `gen.ambiguity`:
   - **unique** — exactly one ground-truth entity satisfies the query.
   - **disambiguated** — unique once the discriminator the ground truth implies is *carried in the
     query* (the preferred repair): `parameter` keeps the **operator name** ("default of *Blur TOP*'s
     Filter Size"); `operator_lookup` appends the **family** when a capability is ambiguous open but
     unique within its family. Real consumer queries carry this context.
   - **multi** — genuinely many valid answers even with context → emit an **accept-set OR-predicate**
     (e.g. a python method on N classes), scored by **SET-recall**, never folded into single-gold MRR.
   Candidates that are ambiguous and not repairable/enumerable are **skipped**.

Offline (JSON/YAML only — never the search stack; `--paraphrase` reads a committed cache, no live LLM)
and **deterministic per `--seed`**. `gen_coverage.py` **asserts per-category unambiguous-rate ≥ 0.9**
(unique+disambiguated / non-multi) and prints it. Cross-cutting cleanup: UTF-8/NFKC normalize +
smart-quotes→ASCII (kills `gPU`/`dMX`/U+FFFD), strip leading `OS:`/`Hardware:`/`License:` boilerplate
and licensing caveats, reject truncated/dangling/`(e.g`-broken queries, de-dupe by query string, skip
deprecated ops. `python_class` enters a gold only if **unique-to-one-op + well-formed** (`*_Class`, no
space) — shared base labels (`Peer Class`, `PanelCOMP_Class` mislabels) would over-match; `op_name_any`
always carries identity. A search-free `--check-kb` confirms every gold matches ≥1 chunk.

Recipes (gold ← named ground-truth source): `operator_lookup` (op summary, name/family/class stripped
→ python_class/op_name); `parameter` (op + **op-SPECIFIC** param — Common-page / >40-op shared params
rejected — **op name kept in the query** → that op's `parameter_group`; tuplets point at the resolved
sub-field `lag1`); `palette_discovery` (comp **distinctive `purpose`** — use_cases & placeholder/dup
purposes rejected — name stripped → block name); `python` (`class_method` desc, method noun kept;
1-class → unique, N-class → multi accept-set); `concept` (glossary definition, term stripped; MediaWiki
namespaces / video-tutorial titles / op-name / near-dup terms blacklisted); `build_instruction` (family
converter → natural "wire/connect/convert a `<src>` into a `<dst>`", incl. POP edges); `negative`
(everyday **+ TD-adjacent HARD** negatives — Shadertoy/Unreal/Houdini/Max — so abstention tests score
calibration, not lexical distance). **howto / recipe / pattern stay hand-authored.**

**Optional `--paraphrase`** (naturalness only; gold NEVER changes): applies `eval/paraphrase_cache.json`
(`'<seed>:<id>' → text`, committed, byte-stable), re-running EVERY gate on each paraphrase (gold
resolves, no leak, still unique to the same gold, not malformed) and falling back to the deterministic
template on any failure. Reports kept/total pass-rate per category. Default OFF — the template set ships.

`coverage_report.py` adds the per-FAMILY / per-CHUNK-TYPE / per-PARAM-KIND tables `run_eval` doesn't
(joins `details.json` on `id` with each item's `gen` block; search-free), reports the **multi slice as
set-recall separately**, and splits negative abstention **easy vs hard**. The numbers reflect whatever
`--backend`/branch `run_eval` used — pass `--stack` to label it. **Measure on the Phase-2 branch**
(`claude/sweet-benz-45506d`, which has `retrieval_stack.py`) for the real with-stack breadth table; a
main-based branch only has the baseline dense path. (`run_eval` pins threads=1 for determinism; set
`EVAL_THREADS=8` to speed the single-trial rerank ~8× on a throwaway measurement worktree.)

### Coverage-maximization — greedy set-cover (Phase 0.6c)

The default mode is **coverage-driven, not fixed-count**: `operator_lookup` ENUMERATES all 663 operators
(no cap) and emits one for each that passes the gates; the rest are SKIPPED with a recorded reason. The
`parameter` recipe is a **greedy set-cover**: Phase A recovers operators NOT covered by `operator_lookup`
(1 op-specific param each → extends operator coverage), Phase B fills every family×kind cell to
`--fk-min`; caps are `--per-op-params` + `--n-parameter`, with a global `--max-total`. **Operator coverage
= an op appears as a gold in operator_lookup ∪ parameter ∪ build.** Seed-0: **≥90% of operators in EVERY
family** (CHOP 94.9 / TOP 95.4 / SOP 96.5 / DAT 91.9 / COMP 95.2 / MAT 100 / POP 99), total 1107.

`gen_coverage.py` writes **`coverage_meta.json`** beside the queries: the full gen-side coverage
accounting + a printed **GOAL CHECKLIST** (operators ≥90% every family · all 6 param kinds · all targeted
chunk types · unambiguous ≥0.9 · 0 answer-leakage · ~100% KB-matchable gold · total ≤ cap) + the
**skipped-operators list with reasons** (a KB-quality finding: ops with no derivable query are almost all
deprecated/removed, guide pages mislabeled as ops, or generic Null/In/Out ops). `coverage_report.py` reads
it and prints the coverage table + checklist **with the measurement OPTIONAL** (`--details`/`--baseline`
omitted ⇒ coverage-only report) — so the breadth goal is verifiable without a (slow, teardown-prone) run.

**`minimal_cover.py` — the FLOOR (fewest tests for solid coverage).** Complement to the max set: greedy
partial set-cover picks the fewest of the 1107 hitting a floor. Because **each test covers exactly one
operator**, ≥80% of ~670 ops needs **≥536 tests (a provable lower bound greedy hits exactly)**. Reports two
floors: **V1** ≥80% ops overall + ≥75% per family (categories free) = **536** (binds on overall-80%); **V2**
the literal ask, + every category ≥75% of its breadth = **774** (binds on the non-operator categories —
python/concept/build/palette). Writes the V2 set to `coverage_minimal.jsonl` (a leaner ship / faster A/B
set), sweeps `--seeds` for the greedy min, deterministic per seed; the MINIMAL section is appended to
`COVERAGE.md`.

### Three-tier eval discipline

| Tier | File | Tests | Decision role |
|---|---|---|---|
| **Frozen 78** | `labeled_queries.jsonl` | the same 23 ops / §3 seeded failures every phase | cross-phase **TREND** gate (don't let it move) |
| **Held-out** | `heldout_queries.jsonl` | paraphrases of the 78 (same predicates) | **OVERFIT** detector / real decision metric |
| **Coverage** | `coverage_queries.jsonl` | breadth across 7 families × §6 chunk types × param kinds; rotates per `--seed`; labels valid by construction | does family X retrieve **at all** |

Coverage = **BREADTH** (does family X retrieve), held-out = **DIFFICULTY** (paraphrase robustness) —
keep both. The coverage tier is a **Phase-3 input**: the embedder A/B is judged on **held-out +
coverage**, NOT the frozen 78.

## Name-integrity gate

For every returned chunk that asserts an operator identity, the gate checks it against
ground truth (`operators.json` + `operator_ground_truth/operator_types.json`):

- **unresolved** — asserts an operator that exists in neither file (hard violation).
- **retokenized** — resolves to a real operator but surfaces the underscored wiki-title
  display name (`Ableton_Link_CHOP`, `GLSL_TOP`) instead of the canonical spaced name.
  This is the plan §3 hazard; the v0.2 target is **0**.

Baseline today: **0 unresolved, 294 retokenized** of 345 identity-bearing returns —
i.e. mangled names dominate, as documented.
