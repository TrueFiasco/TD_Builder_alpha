# TD Builder v0.2 — Phase 0 evaluation harness

The gate every later phase of the KB/retrieval redesign must beat. It measures the
**current shipped search** offline and in-process, with a swappable backend so the
**same harness is reused unchanged through Phase 2/3**.

See the plan §3 (baseline), §5 (ground-truth + name-integrity), §9 Phase 0, §10
(verification): `plan-mode-task-redesign-serialized-kite.md`.

## Files

| File | What it is |
|---|---|
| `labeled_queries.jsonl` | 60 hand-labeled queries (12 × 5 categories). Each row `{id, query, category, relevant_predicate}`. |
| `predicates.py` | Ground-truth loader, predicate matcher, name-integrity gate (reusable). |
| `run_eval.py` | Driver: builds the shipped adapter, runs queries, computes metrics, writes reports. |
| `baseline.json` | The captured current numbers (the gate). Regenerated by `run_eval.py`. |
| `tool_coverage.py` | Phase-0.5 driver for the OTHER KB tools (get_operator_info / get_parameter_detail / find_operator_examples / find_operator_combination / get_network_patterns). |
| `tool_baseline.json` | Captured tool-behaviour baseline. Regenerated by `tool_coverage.py`. |

Staged outputs (NOT committed; reviewer copies) land in
`New KB build/Output/eval/`: `baseline.json`, `BASELINE.md`, `details.json`.

## Run

```bash
py -3.11 eval/run_eval.py                  # both backends, median of 3 trials; writes baseline.json + staged reports
py -3.11 eval/run_eval.py --backend enhanced --repeats 5
py -3.11 eval/run_eval.py --repeats 1      # quick single trial
py -3.11 eval/run_eval.py --kb "C:/path/to/KB" --k 5 --ndcg-k 10
```

Fully offline (local MiniLM + Chroma, no API, cache disabled; the harness forces
`HF_HUB_OFFLINE` + single-thread BLAS for reproducible embeddings). The worktree has
no KB (it's gitignored), so the harness auto-resolves the **main-tree** `KB/` by
stripping the `.claude/worktrees/<name>` tail; override with `--kb` or `EVAL_KB_ROOT`.
Code is imported from this repo's `MCP/server_core`; only the KB **data** comes from
the main tree.

### Reproducibility (`--repeats`)

Embeddings, DB content, and the on-disk HNSW index are all verified byte-stable, but
**ChromaDB's HNSW approximate search is not bit-deterministic across process loads** —
the in-memory graph traversal returns slightly different borderline neighbors, so ~1
rank-5/6 result can flip run-to-run. The jitter is fixed *per process* (in-process
repeats agree), so `--repeats N` runs **N independent subprocesses** (fresh index load
each) and reports the **median** per metric plus the observed band. Measured jitter is
confined to **operator_lookup** (recall@5 0.83–0.92); palette/howto/parameter/python and
`unresolved=0` are invariant. Default `--repeats 3`; use 5 for the canonical capture.

## Backend swap (A/B) — the whole point

`run_eval.py` imports the real `UnifiedSearchAdapter` and calls `.search()` — the exact
call the live `hybrid_search` MCP tool makes.

- `--backend enhanced` → `use_legacy=False`: the **shipped path** and the slot Phase 2
  plugs its new stack into (`_enhanced_search`). **This is THE baseline.**
- `--backend legacy` → `use_legacy=True`: the frozen `HybridGraphRAG` reference; it does
  **not** change as `_enhanced_search` is edited.

Today both produce identical `semantic_results` (same dense MiniLM `TDDocSearch` over
`td_unified`), so their ranking metrics match; they diverge once Phase 2 lands. Re-run
`--backend enhanced` after each Phase-1 section rebuild and after the Phase-2 stack
change to get the attributable delta.

## Metrics

- **recall@k** — hit-rate / success@k: 1 if ≥1 relevant chunk in top-k, else 0.
  (Predicates label an open set, not an enumerated gold set, so this is what the plan's
  "recall@5" table reports.)
- **MRR** — 1 / rank of the first relevant chunk.
- **nDCG@n** — binary-gain DCG over top-n ÷ ideal DCG (all found-relevant first).

## Labels survive re-chunking

A `relevant_predicate` is a disjunction of conjunctive clauses over **STABLE** identity
fields only — never chunk ids:

```json
{"clauses": [{"python_class_any": ["noiseTOP_Class"]},
             {"op_name_any": ["Noise TOP"]}]}
```

A chunk is relevant iff it matches **any** clause; a clause matches iff **all** its keys
match. Supported keys (see `predicates.FIELD_MATCHERS`): `type_any`, `family_any`,
`store_any`, `python_class_any`, `class_any`, `method_any`, `param_code_any`,
`meta_name_any`, `term_any`, `op_name_any` (operator-name, normalized across
name/operator/operator_name), and `text_contains_any` (content substring). Most palette
and howto predicates also carry the **future** v0.2 chunk types (`block_overview`,
`recipe`, `guide`, …) so the same labels score the rebuilt KB without edits.

Labels were hand-verified against the live MCP `hybrid_search` and the ground-truth
files (`KB/operators.json`, `operator_ground_truth/operator_types.json`): operator
`python_class` tokens, parameter codes, palette block names, and `class_method` /
`python_pattern` names all exist as written.

## Tool-coverage harness (`tool_coverage.py`, Phase 0.5)

`run_eval.py` gates the *ranking* tool (`hybrid_search`). `tool_coverage.py` gates the
**other** KB-dependent MCP tools — the hydration/lookup + pattern tools the builder uses
once it knows an operator — so tool regressions/fixes are visible as the KB is rebuilt. It
drives the real engine the MCP handlers delegate to (`UnifiedGraphQuery`), offline +
in-process, reusing the same labels, and prints a compact scorecard (details → disk).

```bash
py -3.11 eval/tool_coverage.py            # writes tool_baseline.json + New KB build/Output/eval/TOOL_COVERAGE.md
```

Checks: `get_operator_info` (resolve / canonical name / has-params), `get_parameter_detail`
(param found, default present, **default matches the `operator_ground_truth` live-TD
capture**), `find_operator_examples` (operators with ≥1 example), `find_operator_combination`
(intent operator-sets, natural tokens), `get_network_patterns` (inventory + `DAT:text`
readMe-pollution count). Deterministic (graph/dict lookups, no HNSW jitter). The defaults
check is the headline: it tells you which param defaults the tool already gets right (so §6.2
re-grounding can target the long tail) vs wrong.

Captured baseline: get_operator_info 15/15 resolve correctly; get_parameter_detail returns
the correct live-TD default on **12/12 verifiable** common params (better than the plan
assumed — the structured `default` field is right; only the description prose has `⊞`
artifacts); find_operator_examples 14/15; find_operator_combination 4/8 intents with natural
tokens (misses are TD short-OPType token brittleness — comp/geo/cam — not absent data);
get_network_patterns 45/45 readMe-polluted. All cross-verified against the live MCP server.

## Name-integrity gate

For every returned chunk that asserts an operator identity, the gate checks it against
ground truth (`operators.json` + `operator_ground_truth/operator_types.json`):

- **unresolved** — asserts an operator that exists in neither file (hard violation).
- **retokenized** — resolves to a real operator but surfaces the underscored wiki-title
  display name (`Ableton_Link_CHOP`, `GLSL_TOP`) instead of the canonical spaced name.
  This is the plan §3 hazard; the v0.2 target is **0**.

Baseline today: **0 unresolved, 294 retokenized** of 345 identity-bearing returns —
i.e. mangled names dominate, as documented.
